# F021.1 — Update banner (service-worker lifecycle)

Parent: **F021 — PWA in-app update** (mechanism + fleet rationale in the epic plan-doc).

## Files

- `scripts/stamp-sw.cjs` — npm `prebuild` step; stamps `SW_BUILD` in `public/sw.js` with a per-build token so each deploy ships a byte-changed worker.
- `public/sw.js` — adds `SW_BUILD`/`CACHE` token; **removes `skipWaiting()` on install** (user-gated); adds a `message` handler that `skipWaiting()`s on `{type:'SKIP_WAITING'}`.
- `components/pwa-update-bar.tsx` — detects a waiting worker (`registration.waiting` at mount + `updatefound`→`installed`, guarded on an existing controller); re-checks via `reg.update()` on 2-min interval + focus + visibilitychange; Update → `SKIP_WAITING` → `controllerchange` → single reload (updatingRef-guarded, 1.5s backstop); Later dismisses; renders `null` when nothing is waiting.
- `lib/pwa.ts` — `forcePwaUpdate()` (clear caches + reload) for the user-menu "Opdater app" manual refresh.
- `app/(app)/layout.tsx` — renders `<PwaUpdateBar/>`.
- `components/nav-user.tsx` — "Opdater app" menu item.

## Superseded

The initial build-id `/api/version` polling approach (first commit) was replaced with this SW-lifecycle mechanism after consulting fds + cardmem, for fleet consistency and to avoid a version-bookkeeping endpoint. `/api/version` was removed.
