# F021.1 — Update banner (via @broberg/pwa)

Parent: **F021 — PWA in-app update**.

## Final implementation — consumes the shared package

The SW-lifecycle detection/handshake was first hand-rolled here, then flagged to `components` as the third fleet copy (fds + cardmem + pitch). `components` shipped **`@broberg/pwa@0.1.0`**; pitch-vault is its first live consumer.

- `components/pwa-update-bar.tsx` — our styled "Update available" card only. Detection comes from `usePwaUpdate({ register: false, pollIntervalMs: 120_000, disabled: NODE_ENV !== 'production' })` (`@broberg/pwa/react`). The hook owns `registration.waiting`/`updatefound` detection, first-install suppression, the focus/visibility + interval poll, and `SKIP_WAITING → controllerchange → one guarded reload`. `Later` uses local dismiss state; `Update` → `applyUpdate()`.
- `register: false` because `components/sw-register.tsx` still registers `/sw.js` app-wide (root layout) — the hook attaches via `navigator.serviceWorker.ready` instead of double-registering.
- **Stays ours (the package does not stamp the SW):** `public/sw.js` SKIP_WAITING handler (identical contract) + `scripts/stamp-sw.cjs` per-deploy byte-stamp — the byte-diff is what makes `reg.update()` install a new worker.
- `lib/pwa.ts` `forcePwaUpdate()` (clear caches + reload) remains the user-menu "Opdater app" manual escape hatch.

Removed: all hand-rolled `useEffect`/refs/`controllerchange`/poll — now the package's responsibility, not drift waiting to happen.
